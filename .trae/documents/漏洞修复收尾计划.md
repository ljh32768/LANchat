### 1. V5 补全：TCP 空闲超时 [P1 补全]

**文件**：`pages/src/main/network/network.js`

**问题**：已有 `setKeepAlive(true, 30000)` 和 `setNoDelay(true)`，但缺 `setTimeout(60000)` + timeout 事件处理。V4 心跳可补偿，但显式超时是纵深防御。

**改动**：在 3 处 TCP socket 创建点（`_onHostSocket` ~L414、`joinSession` ~L548、`_reconnectJoined` ~L651）的 `setKeepAlive`/`setNoDelay` 之后追加：
```javascript
socket.setTimeout(60000);
socket.on('timeout', () => {
  console.warn('[net] tcp idle timeout, destroy', socket.remoteAddress);
  socket.destroy();
});
```

注意：`_onHostSocket` 的 timeout 会触发 `close` 事件 → 已有 member 清理逻辑；`joinSession`/`_reconnectJoined` 的 timeout 会触发 `end`/`error` → 已有重连逻辑。无需额外处理。

### 9. V19：_broadcastPresence 防抖 [P3 性能]

**文件**：`pages/src/main/network/network.js`

**问题**：`_broadcastPresence` 在 join/leave/close/setClientInfo 等操作时直接调用，快速连续操作触发密集广播。

**改动**：在类中增加防抖包装：
```javascript
// constructor 中
this._broadcastTimer = null;

// 新增防抖方法（替代直接调用 _broadcastPresence 的场景）
_broadcastPresenceDebounced() {
  if (this._broadcastTimer) return;
  this._broadcastTimer = setTimeout(() => {
    this._broadcastTimer = null;
    this._broadcastPresence();
  }, 500);
}
```
将 `hostSession` 成功后、`closeHostedSession`、`leaveSession`、`setClientInfo` 等主动触发广播的调用点改为 `_broadcastPresenceDebounced()`。

注意：`start()` 中的定时广播（`setInterval(() => this._broadcastPresence(), ...)`）和 `_broadcastPresence(sessionEnd)` 带参数的会话结束广播保持直接调用（不防抖，确保时效性）。

### 10. V25：CSP 改进 [P3 安全]

**文件**：`pages/src/renderer/index.html`（L5）

**问题**：CSP 含 `'unsafe-inline'`，削弱 XSS 防护。

**改动**：拆分为细粒度指令，将 `'unsafe-inline'` 仅保留给 `style-src`（React 内联样式需要），`script-src` 移除 `'unsafe-inline'`，新增 `object-src 'none'` 和 `base-uri 'self'`：
```html
<meta http-equiv="Content-Security-Policy" content="
  default-src 'self' data: blob:;
  script-src 'self';
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: blob:;
  connect-src 'self' ws://localhost:5173 http://localhost:5173;
  object-src 'none';
  base-uri 'self';
" />
```

**风险**：Vite 生产构建若注入 inline script（如 module preload polyfill），`script-src 'self'` 会导致阻断。修改后需测试 `npm run dev` 和 `npm run build` 两种模式。若构建后报 CSP 违规，可改用 nonce 方案或回退保留 `script-src 'self' 'unsafe-inline'`。

1. 安装依赖：`npm install react-window`

```javascript
import { FixedSizeList as List } from 'react-window';

// 消息项高度（固定估算，文本超长时由 CSS overflow 处理）
const ITEM_HEIGHT = 72;

// 虚拟列表行渲染器
const Row = ({ index, style }) => {

// 替换原 messages.map 为：
```

3. 自动滚动到底部逻辑调整：用 `listRef.current.scrollToItem(messages.length - 1)` 替代 `scrollRef.current.scrollTop = ...`。

2. `database.js` 追加清理函数：
```javascript
function cleanupOldMessages(daysThreshold = 30) {
  const cutoff = Date.now() - daysThreshold * 24 * 60 * 60 * 1000;
  // 先删除关联的文件记录（外键 CASCADE 会自动处理，但 sql.js 需手动）
  execute('DELETE FROM files WHERE message_id IN (SELECT message_id FROM messages WHERE timestamp < ?)', [cutoff]);
  execute('DELETE FROM messages WHERE timestamp < ?', [cutoff]);
  // 清理无消息引用的会话（已结束且无消息）
  execute("DELETE FROM sessions WHERE status = 'ended' AND session_id NOT IN (SELECT DISTINCT session_id FROM messages)");
  // VACUUM 回收空间
  if (db) db.run('VACUUM;');
  persist();
  return { deleted: true, cutoff };
}
```

3. `ipc/index.js` 追加 handler：
```javascript
ipcMain.handle(IPC.DB_CLEANUP, (_e, { days }) => {
```

### 13. V28：版本化数据库迁移 [P3 可维护性]

**文件**：`pages/src/main/db/database.js`（L94-102 现有迁移逻辑）

**问题**：现有迁移用 `PRAGMA table_info` 检测列是否存在，无版本号管理，多步迁移难以追踪。

**改动**：用 `PRAGMA user_version` 实现版本化迁移：

```javascript
// 替换 L94-102 的简易迁移逻辑
function runMigrations() {
  const versionRow = queryOne('PRAGMA user_version');
  let version = versionRow?.user_version || 0;

  // v1: 添加 sessions.last_activity_at 列
  if (version < 1) {
    try {
      const cols = query("PRAGMA table_info(sessions)");
      if (cols.length > 0 && !cols.some((c) => c.name === 'last_activity_at')) {
        execute('ALTER TABLE sessions ADD COLUMN last_activity_at INTEGER');
      }
    } catch (e) {
      console.error('[db] migration v1 failed:', e);
    }
    version = 1;
  }

  // 未来迁移在此追加 if (version < 2) { ... version = 2; }

  // 写入版本号
  if (db) db.run(`PRAGMA user_version = ${version};`);
}
```

在 `initDatabase()` 中 `db.run(SCHEMA_SQL)` 之后、`persist()` 之前调用 `runMigrations()`。

## 假设与决策

1. **V14 使用 FixedSizeList**：消息高度固定为 72px，CSS 限制气泡最大高度。若体验不佳，后续可升级为 VariableSizeList。这是最简方案，避免引入复杂的动态高度测量。
2. **V25 保留 style-src 'unsafe-inline'**：React 运行时会注入内联样式（如 style 属性），移除会导致样式失效。script-src 移除 'unsafe-inline' 是主要改进点。
3. **V19 仅防抖主动广播**：定时广播（3s 间隔）和会话结束广播保持直接调用，确保时效性。仅对操作触发的即时广播做防抖。
4. **V18 清理策略为手动触发**：不自动清理（避免删除用户可能需要的历史），由用户在设置中主动执行。30 天为默认阈值。
5. **V28 版本从 1 开始**：当前数据库结构视为 v1 基线，`last_activity_at` 列添加为 v1 迁移内容。
6. **V21 仅做错误恢复**：不改变 await-first 核心模式，仅补全失败时恢复输入文本 + 错误日志。
