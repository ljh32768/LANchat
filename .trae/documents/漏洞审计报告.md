## 一、致命漏洞（必须立即修复）

### V1 通用 SQL 通道暴露给渲染进程 — 任意数据库操纵
- **严重程度**：致命
- **清单项**：1.2 / 1.5
- **触发条件**：`preload.js:10` 白名单含 `'db:query'`、`'db:execute'`；`ipc/index.js:433-434` 直接转发任意 SQL
- **后果**：渲染进程（或 XSS 注入脚本）可执行 `DROP TABLE`、`DELETE FROM messages`、读取全部联系人/消息、篡改 settings。配合 CSP `'unsafe-inline'`（V25）形成可被 XSS 放大的攻击链
- **修复方案**：移除 `preload.js` 白名单中的 `'db:query'`、`'db:execute'`，删除 `ipc/index.js:433-434` 的两个处理器。已有 `message:list`、`contacts:list` 等具体业务通道可覆盖所有合法需求

### V2 文件路径遍历漏洞 — 任意文件写入/读取
- **严重程度**：致命
- **清单项**：2.8
- **触发条件**：`network.js:655` `const dest = path.join(this.fileStoreDir, header.file_id)` — `file_id` 由客户端 TCP 包控制，未校验格式。`network.js:683` 下载路径同理
- **后果**：构造 `file_id='..\\..\\..\\Windows\\System32\\evil.dll'` 可覆盖系统文件（需进程权限）；任何能连到 47890 端口的人可越权下载任意暂存文件（`_serveFileDownload` 不校验请求方是否为会话成员）
- **修复方案**：
  ```javascript
  // network.js 上传/下载前校验 file_id 为 UUID 格式
  const UUID_RE = /^[a-f0-9-]{36}$/i;
  if (!UUID_RE.test(file_id)) { socket.destroy(); return; }
  const dest = path.join(this.fileStoreDir, file_id);
  // 二次校验解析后路径仍在 fileStoreDir 内
  const resolved = path.resolve(dest);
  if (!resolved.startsWith(path.resolve(this.fileStoreDir) + path.sep)) {
    socket.destroy(); return;
  }
  ```
  下载时额外校验请求方 socket 是否为已加入会话的成员

### V3 身份冒充零成本 — client_id 完全自报无校验
- **严重程度**：致命
- **清单项**：2.9 / 3.1 / 3.2
- **触发条件**：`network.js:388-416` `_onHostLine` 直接信任 `msg.client_id`，不与 `socket.remoteAddress` 绑定。UDP PRESENCE 明文广播 `client_id`（`network.js:175-178`），局域网内任何人可嗅探
- **后果**：攻击者获取受害者 `client_id` 后，连接主机 TCP 端口发送 `{kind:'msg', client_id:'受害者ID', ...}`，主机会以受害者身份存储并转发消息。`handleReceivedMessage`（`ipc/index.js:29-81`）和 `_relay`（`network.js:445-452`）均不校验来源
- **修复方案**：
  ```javascript
  // _onHostSocket 接受连接时记录对端 IP
  _onHostSocket(socket, sessionId) {
    const memberInfo = { client_id: null, nickname: null, socket, remoteAddress: socket.remoteAddress };
    // ...
  }
  // _onHostLine 处理 join 时绑定 client_id 与 remoteAddress
  if (msg.kind === 'join') {
    memberInfo.client_id = msg.client_id;
    memberInfo.remoteAddress = socket.remoteAddress; // 锁定
    // 校验该 IP 是否在 peers 中且 client_id 匹配
    const peer = this.peers.get(msg.client_id);
    if (peer && peer.ip !== socket.remoteAddress) {
      console.warn('[net] client_id IP mismatch, reject');
      socket.destroy(); return;
    }
  }
  // _onHostLine 处理 msg 时，sender 一律取 memberInfo.client_id 而非 msg.sender_contact_id
  if (msg.kind === 'msg') {
    msg.sender_contact_id = memberInfo.client_id; // 强制覆盖
    msg.sender_ip = memberInfo.remoteAddress;
    this.emit('message', msg);
    this._relay(sessionId, msg, socket);
  }
  ```

## 二、严重漏洞

### V4 主机异常离线客户端无法清理 — 会话永久挂起
- **严重程度**：严重
- **清单项**：2.4 / 2.6
- **触发条件**：主机断电/kill -9/拔网线时，TCP 无 keep-alive（V5）、无应用层心跳。`_sweep`（`network.js:310-324`）只清理 `peers` 和 `discoveredSessions`，不清理 `joined` 会话
- **后果**：客户端 UI 长时间显示"已加入会话"且能发消息（消息进 TCP 缓冲区但永远发不到对端），用户感知不到对方已离线
- **修复方案**：
  1. `_sweep` 中增加 joined 会话清理：若 `joined` 会话的 host 不在 `peers` 中（或 peer 超时），触发重连放弃
  2. TCP 层加心跳：`socket.setKeepAlive(true, 30000)` + 应用层 PING/PONG（30s 一次，3 次未响应则断开重连）

### V5 TCP 无 keep-alive/超时/Nagle — 连接静默断开
- **严重程度**：严重
- **清单项**：2.3
- **触发条件**：`network.js` 全文无 `socket.setKeepAlive()`、`socket.setTimeout()`、`socket.setNoDelay()` 调用
- **后果**：NAT 表项超时/对端 kill -9 时长时间不触发 `end`/`error`；连接阶段若对端不响应 SYN-ACK 依赖 OS 默认（~21-75s）无主动超时
- **修复方案**：
  ```javascript
  // joinSession 和 _onHostSocket 中
  socket.setKeepAlive(true, 30000);  // 30s keep-alive
  socket.setTimeout(60000);          // 60s 空闲超时
  socket.setNoDelay(true);           // 禁用 Nagle，消息实时性
  socket.on('timeout', () => { socket.destroy(); });
  ```

### V6 sql.js 全同步阻塞主进程 — 随 DB 增长劣化
- **严重程度**：严重
- **清单项**：4.1 / 1.3
- **触发条件**：`database.js:90-103` `query()` 全同步；`database.js:15-23` `persist()` 每次 `execute` 后 `db.export()`（序列化整个内存库 O(DB体积)）+ `fs.writeFileSync` 同步写盘。IPC 处理器在主线程直接调用
- **后果**：每条消息写入延迟随 DB 体积线性增长，主进程事件循环被阻塞，UDP/TCP 处理延迟，UI 卡顿
- **修复方案**：
  1. 短期：`persist()` 改为防抖（如 500ms 内多次写入只持久化一次），避免每条消息都全量导出
  2. 中期：`persist()` 用 `write-to-temp-then-rename` 原子写入（同时解决 V8）
  3. 长期：迁移到 `better-sqlite3`（原生同步但 C++ 实现，性能高 10-100 倍）或 `sqlite3`（异步）

### V7 文件传输无超时清理 — socket/文件句柄泄漏
- **严重程度**：严重
- **清单项**：2.7
- **触发条件**：`network.js:638-680` 上传、`network.js:682-695` 下载、`network.js:698-714` 客户端上传、`network.js:717-751` 客户端下载 — 均无 `setTimeout`。中途断流但 TCP 未报错时，`state.ws` 和 `socket` 一直挂着
- **后果**：文件句柄不释放，`state.remaining` 永远 > 0，长时间运行累积泄漏
- **修复方案**：所有文件传输 socket 加 `setTimeout(120000)`（2 分钟），超时后 `socket.destroy()` + `state.ws.destroy()` + 回调错误

### V8 `session:created` 通道未在 preload 白名单 — 私聊会话不刷新
- **严重程度**：严重
- **清单项**：5.2 / 5.4
- **触发条件**：`preload.js:17-25` `eventChannels` Set 不含 `'session:created'`。`preload.js:37` `if (!eventChannels.has(channel)) return;` 直接返回 undefined，监听器从未注册
- **后果**：收到私聊邀请时主进程自动创建本地会话并发送 `SESSION_CREATED` 事件，但渲染层 `useIpcEvents.js:78` 的处理函数**永不执行**。用户不会在侧边栏看到新创建的私聊会话，直到其他操作触发 `load()`
- **修复方案**：`preload.js:17-25` 的 `eventChannels` 数组添加 `'session:created'`

### V9 sandbox: false — 攻击面扩大
- **严重程度**：严重
- **清单项**：1.1
- **触发条件**：`main.js:48` `sandbox: false`。据项目记忆，是为在开发环境把 userData 重定向到项目内 `data/` 目录而关闭
- **后果**：preload 脚本拥有完整 Node.js 能力，一旦 preload 被污染或白名单通道越权（如 V1），攻击面显著扩大
- **修复方案**：生产环境启用 sandbox（`main.js:48` 改为 `sandbox: !isDev` 或 `sandbox: true`）。开发环境的 userData 重定向在 `app.whenReady()` 后设置即可，不需要关闭 sandbox

## 三、主要漏洞

### V10 UDP 包无大小/schema 校验 — 拒绝服务
- **严重程度**：主要
- **清单项**：2.2
- **触发条件**：`network.js:208-278` `_onUdpMessage` 无 `msg.length` 上限，无字段 schema 校验，无未知 type 告警
- **后果**：攻击者发送超大 UDP 包，`JSON.parse` 同步阻塞主进程；畸形 `sessions` 数组污染 `discoveredSessions`
- **修复方案**：
  ```javascript
  _onUdpMessage(msg, rinfo) {
    if (msg.length > 8192) { console.warn('[net] oversized udp packet'); return; }
    // ... 现有 JSON.parse
    if (!['presence','session_end','private_invite'].includes(packet.type)) {
      console.warn('[net] unknown packet type', packet.type); return;
    }
  }
  ```

### V11 文件端口无冲突重试 — 文件传输静默失效
- **严重程度**：主要
- **清单项**：2.1
- **触发条件**：`network.js:632-636` `_startFileServer` 直接 `listen(47890)`，`fileServer.on('error')` 只打日志，无 EADDRINUSE 重试（对比 `hostSession` 有 20 次重试）
- **后果**：47890 被占用时文件传输功能静默失效，用户无感知
- **修复方案**：`_startFileServer` 仿照 `hostSession` 加 EADDRINUSE 重试，或动态分配端口并记录到 `this.fileServerPort`

### V12 persist() 非原子写入 — 崩溃损坏 DB
- **严重程度**：主要
- **清单项**：4.2
- **触发条件**：`database.js:19` `fs.writeFileSync(dbPath, Buffer.from(data))` 直接覆盖原文件，未用 write-to-temp-then-rename
- **后果**：写入过程中断电/崩溃，`chat.db` 停留半写入状态，下次启动解析失败
- **修复方案**：
  ```javascript
  function persist() {
    const tmp = dbPath + '.tmp';
    fs.writeFileSync(tmp, Buffer.from(db.export()));
    fs.renameSync(tmp, dbPath); // 原子替换
  }
  ```

### V13 昵称后端无校验 — 输入层防御缺失
- **严重程度**：主要
- **清单项**：3.3
- **触发条件**：`ipc/index.js:195-199` `CLIENT_SET_NICKNAME` 直接 `db.setNickname(nickname)` 无长度/字符集/trim 校验。远端消息的 `sender_nickname` 同样未校验
- **后果**：DevTools 或直接 invoke 可绕过前端 maxLength=24 限制。当前 React 转义防住 XSS，但纵深防御缺失
- **修复方案**：
  ```javascript
  function sanitizeNickname(s) {
    s = String(s || '').trim().slice(0, 24);
    return s.replace(/[\r\n<>]/g, ''); // 去除换行和尖括号
  }
  // CLIENT_SET_NICKNAME 和 handleReceivedMessage 中统一调用
  ```

### V14 无虚拟滚动 — 长会话 DOM 爆炸
- **严重程度**：主要
- **清单项**：6.3
- **触发条件**：`ChatView.jsx:70-86` `messages.map()` 全量渲染，无虚拟化库
- **后果**：数千条消息时 DOM 节点线性增长，滚动和渲染性能显著下降。每条消息含嵌套 div+span+CSS 动画
- **修复方案**：引入 `react-window` 或 `@tanstack/react-virtual`，仅渲染可视区域消息（约 20-30 条）

### V15 高频事件无节流 — 过度重渲染
- **严重程度**：主要
- **清单项**：5.3
- **触发条件**：`useIpcEvents.js:62-64` `file:download-progress` 每个 TCP 块触发 `setProgress`；`useIpcEvents.js:25-36` `presence-update` 每个 peer 每 3s 触发
- **后果**：大文件下载时每秒数十次 React 重渲染；N 个 peer 时每 3s N 次 `setOnline`（创建新 Set + 新数组）
- **修复方案**：`file:download-progress` 用 `requestAnimationFrame` 节流（或 lodash.throttle 100ms）；`presence-update` 批量累积后单次 flush

### V16 文件上传/下载按钮缺少 in-flight 禁用 — 重复触发
- **严重程度**：主要
- **清单项**：6.6
- **触发条件**：`MessageInput.jsx:28` 文件上传按钮 `disabled={disabled}` 仅按会话 ended 禁用，上传期间可再次点击；`MessageBubble.jsx:91` 下载按钮无 disabled
- **后果**：并发上传/下载，资源浪费
- **修复方案**：上传/下载期间根据 `useFilesStore` 的 status 设为 `'uploading'`/`'downloading'` 时禁用按钮

### V17 SQLite 外键约束未启用 — 数据完整性无保证
- **严重程度**：主要
- **清单项**：4.4（关联）
- **触发条件**：`schema.js` 声明了 FOREIGN KEY，但 `database.js` 从未执行 `PRAGMA foreign_keys = ON;`
- **后果**：外键约束不生效，可插入孤儿消息记录（session_id 不存在）
- **修复方案**：`initDatabase()` 中 `db.run(SCHEMA_SQL)` 前加 `db.run('PRAGMA foreign_keys = ON;')`

## 四、次要漏洞

### V18 无数据库清理/VACUUM — DB 无界增长
- **严重程度**：次要
- **清单项**：4.3
- **触发条件**：无 retention policy、无 VACUUM、`leaveSession` 保留消息
- **后果**：长期使用 DB 文件体积只增不减
- **修复方案**：设置中加"清理 N 天前消息"入口；定期 `VACUUM`

### V19 事件触发广播无节流
- **严重程度**：次要
- **清单项**：3.4
- **触发条件**：`_broadcastPresence()` 在 join/leave/close/setClientInfo 时直接调用，无 debounce
- **后果**：快速 join/leave 或频繁改昵称触发密集广播
- **修复方案**：`_broadcastPresence` 包 debounce（如 500ms）

### V20 渲染层会话状态用魔法字符串
- **严重程度**：次要
- **清单项**：5.4
- **触发条件**：渲染层 `ChatView.jsx:47`、`useSessionsStore.js:51,88,95` 硬编码 `'ended'`，未引用 `SESSION_STATUS` 常量
- **后果**：状态字符串散布，重构易遗漏
- **修复方案**：渲染层也从 `@shared/constants` 导入 `SESSION_STATUS`

### V21 send 乐观更新无回滚
- **严重程度**：次要
- **清单项**：5.4
- **触发条件**：`useMessagesStore.js:22-38` IPC 返回前就加入本地 store，无回滚
- **后果**：IPC 失败时本地出现未实际发送的消息
- **修复方案**：`send` 失败时从 store 移除该消息并提示

### V22 box-shadow 动画未在性能模式关闭
- **严重程度**：次要
- **清单项**：6.4
- **触发条件**：`theme.css:109-111` `body.perf-mode` 仅关闭 borderBreath/scanlineDrift/bubbleIn，未关闭 dotBreath/unreadGlow/inputGlow
- **后果**：性能模式下仍有 box-shadow 动画触发重绘
- **修复方案**：`body.perf-mode` 下禁用所有 box-shadow 动画

### V23 shell.openExternal 未校验协议
- **严重程度**：次要
- **清单项**：1.1（关联）
- **触发条件**：`main.js:77` `setWindowOpenHandler` 对所有 url 调用 `openExternal`，未过滤 `file://`、`javascript:`
- **后果**：恶意链接可触发危险协议
- **修复方案**：
  ```javascript
  if (url.startsWith('http://') || url.startsWith('https://')) {
    shell.openExternal(url);
  }
  ```

### V24 addMessage 非幂等 INSERT — TOCTOU 竞态
- **严重程度**：次要
- **清单项**：4.4
- **触发条件**：`database.js:221-225` 用普通 `INSERT`，`handleReceivedMessage` 前置 SELECT 检查存在 TOCTOU 竞态窗口
- **后果**：并发相同 message_id 时第二个 INSERT 抛 PRIMARY KEY 冲突（被 try-catch 吞掉，但 files 表写入被跳过）
- **修复方案**：`INSERT OR IGNORE` 替代普通 `INSERT`

## 五、建议级改进

### V25 CSP 含 'unsafe-inline'
- **清单项**：10.1
- **实际情况**：`index.html:5` CSP 含 `'unsafe-inline'`，削弱 XSS 防护
- **建议**：用 nonce/hash 替代 `'unsafe-inline'`；加 `object-src 'none'`；主进程通过 `onHeadersReceived` 注入 HTTP 头方式 CSP

### V26 生产环境未禁用 DevTools
- **清单项**：1.1（关联）
- **实际情况**：`main.js:56` 注释说手动 F12，但代码未拦截，生产构建仍可唤出 DevTools
- **建议**：生产环境 `webContents.on('before-input-event')` 拦截 F12/Ctrl+Shift+I

### V27 strictSSL: false — 构建期供应链风险
- **清单项**：9.1（关联）
- **实际情况**：`package.json:55` `electronDownload.strictSSL: false`
- **建议**：可信网络环境下构建，或移除该配置用默认 SSL 校验

### V28 无版本化数据库迁移
- **清单项**：9.4
- **实际情况**：`database.js:63-71` 简易 PRAGMA 检测式迁移，无 `PRAGMA user_version`
- **建议**：引入版本化迁移机制，支持多步有序迁移

## 验证步骤

修复后按以下步骤验证：

1. **V1 验证**：渲染进程调用 `window.api.invoke('db:execute', {sql:'DROP TABLE messages'})` 应返回 `Disallowed invoke channel` 错误
2. **V2 验证**：构造 `file_id='../../evil.exe'` 的 TCP 上传包，应被拒绝且 socket 关闭
3. **V3 验证**：用受害者 client_id 从不同 IP 连接主机发送消息，应被拒绝
