# 局域网聊天室 — 工业级质量与安全性审查清单

## 审查目标

对本项目的**完整代码、架构设计、网络协议、数据流、UI 实现、打包配置**进行全面审计，提前暴露生产环境中可能出现的问题，确保系统稳定、安全、可维护。

------

## 审查范围与方法

+ 假设项目结构如下，需扫描所有相关文件：

  text

  ```
  main.js / preload.js / src/main/* (主进程代码)
  src/renderer/* (React组件、Zustand stores、hooks)
  src/shared/* (常量、协议定义)
  vite.config.js / package.json / electron-builder配置
  ```

  

+ 每个问题项必须给出：

  1. **严重程度**：致命 / 严重 / 主要 / 次要 / 建议
  2. **触发条件**：真实场景或代码路径
  3. **后果**：用户体验、数据安全、系统稳定性
  4. **修复方案**：具体可操作的代码或架构调整

------

## 一、架构与进程隔离

1.1 主进程与渲染进程隔离是否完全遵循 Electron 安全最佳实践？
  检查 `nodeIntegration: false`, `contextIsolation: true`, `sandbox` 等配置。
1.2 `preload.js` 暴露的 API (`window.electronAPI`) 是否为最小必要集？是否有暴露 `ipcRenderer.send` 或 `invoke` 通道可被渲染进程滥用？
1.3 原生模块 `sql.js` 运行在哪一进程？若在主进程，通过 IPC 调用是否存在阻塞事件循环的风险？
1.4 渲染进程是否直接访问了 Node.js API（如 `fs`、`path`、`net`）？
1.5 是否有任一 IPC 通道可传递任意函数、动态代码或文件路径导致远程代码执行？

## 二、网络协议与传输层健壮性

2.1 UDP 广播的端口是否固定？多个实例运行时是否有端口冲突检测与提示？
2.2 UDP 消息的序列化/反序列化是否有严格的边界校验？是否处理了恶意超大包或畸形包？
2.3 TCP 连接是否正确设置了超时、`keep-alive` 和 `Nagle` 算法优化？
2.4 会话心跳包（PING/PONG）是否实现？当连续 3 次未收到 PONG 是否正确触发重连？
2.5 重连逻辑是否为每个断开会话独立运行？是否避免重连风暴（如多个会话同时断开时的定时器爆炸）？
2.6 主机异常离线（进程被杀、断电）时，客户端是否有超时清理机制，而不会永远停在“连接中断”状态？
2.7 文件传输的临时连接是否在传输完成/失败后正确关闭？是否有超时清理？
2.8 文件接收路径是否存在路径遍历漏洞（如 `../../system32/evil.exe`）？是否限制下载目录只能为应用指定文件夹？
2.9 消息接收时是否验证了发送者 IP 与 TCP 连接的对端 IP 一致，防止 UDP 广播伪造然后 TCP 连接欺骗？

## 三、身份认证与信任边界

3.1 客户端 UUID 是否存储在本地明文 JSON/SQLite 中？若被他人复制，是否可彻底冒充身份？是否考虑引入简单的公私钥签名作为未来增强？
3.2 防伪造身份降级显示逻辑是纯前端实现，还是也反映在数据库关联中？如果恶意客户端在通信中篡改了 `contact_id`，后端校验吗？
3.3 用户昵称是否有长度和字符集限制（防 XSS，如 `<script>alert(1)</script>`）？
3.4 联系人信息是否会被恶意广播频繁更新，导致 UI 卡顿或数据库写入放大？是否有频率限制？

## 四、数据持久化与一致性

4.1 `sql.js` 操作是同步还是异步？若在主进程异步 IPC 调用，如何保证消息写入顺序与接收顺序一致？
4.2 应用崩溃或断电时，内存中尚未写入数据库的消息是否会丢失？是否有写入缓冲队列保证最终持久化？
4.3 数据库文件体积无限增长时是否有自动清理策略（如保留最近 N 天记录）或手动清理入口？
4.4 新加入会话时拉取的历史消息是否可能与本机已有消息重复？是否有去重机制（基于 `message_id` 唯一约束）？
4.5 `sql.js` 的 WASM 文件在打包后路径是否正确？是否在 `asarUnpack` 中正确解包？是否有 fallback 提示？
4.6 多实例（测试场景）使用相同数据目录时是否给出明确警告，而不是静默损坏数据库？

## 五、状态管理与数据流

5.1 Zustand store 是否按职责合理拆分（session、message、contact、settings 等）？
5.2 IPC 事件监听器是否在组件卸载时正确移除，避免内存泄漏？
5.3 高频事件（如消息接收、文件下载进度）是否进行了批量更新或节流，避免过多重渲染？
5.4 会话状态机（`active` → `reconnecting` → `ended`）是否在所有代码路径中严格一致，不存在非法状态？
5.5 多个 store 之间的依赖是否清晰，是否使用 `subscribeWithSelector` 防止不必要的重渲染？

## 六、用户界面与性能

6.1 透明窗口和胶囊形状在 Windows 7/8/Linux 上的兼容性如何？是否有 fallback 矩形模式？
6.2 所有动画是否优先使用 `transform` 和 `opacity`？是否通过 Chrome DevTools Performance 面板验证无长任务？
6.3 消息列表是否使用了虚拟滚动或分页加载，防止数千条消息时 DOM 节点爆炸？
6.4 扫描线伪元素和全息边框的 `box-shadow` 动画是否在“性能模式”下被移除或降级？
6.5 是否有统一的错误边界组件（Error Boundary）防止白屏？
6.6 用户网络操作（发送消息、下载文件）时是否有 loading/disable 状态防止重复点击？

## 七、错误处理与日志

7.1 主进程所有 TCP/UDP 错误是否被捕获并记录，而不是让进程崩溃？
7.2 是否实现了一套分级日志（debug/info/warn/error）并输出到文件，方便用户反馈问题时排查？
7.3 渲染进程的未捕获异常和 Promise 拒绝是否通过全局监听器上报？
7.4 数据库操作失败（如磁盘满）时是否有用户提示，而不是静默忽略？
7.5 文件下载失败是否有重试机制？部分下载的文件是否会残留垃圾文件？

## 八、并发与边界场景

8.1 同时收到多个文件下载请求时，下载队列是否串行/并发控制？是否占满带宽影响聊天消息？
8.2 一个用户同时在多个会话中发言，消息推送顺序是否能保证？
8.3 主机在解散会话的同时收到新消息，是否先通知解散后丢弃新消息？
8.4 操作系统睡眠/休眠唤醒后，所有 TCP 连接是否会被重置？是否触发正确的重连逻辑？
8.5 系统时间大幅度调整（如手动改时钟）是否影响消息排序或会话过期判断？

## 九、打包与分发

9.1 `electron-builder` 配置中的 `asarUnpack` 是否解压了 `sql.js` 的 WASM 以及可能用到的其他二进制模块？
9.2 NSIS 安装包在无管理员权限下是否能正常安装到用户目录？是否测试过 `perMachine: false`？
9.3 便携版（zip）或绿色版是否与安装版功能完全一致，数据目录正确隔离？
9.4 版本升级时，旧版本地数据库结构若有变化，是否实现数据库迁移逻辑？
9.5 是否有基本的功能自检（smoke test）确保打包后的应用能启动并显示 UI？

## 十、安全合规（补充）

10.1 所有用户输入（昵称、消息、文件名）输出到 DOM 时是否进行了转义（React 默认转义，但 `dangerouslySetInnerHTML` 禁用）？
10.2 是否有任何第三方依赖存在已知高危漏洞？（可用 `npm audit` 检查）
10.3 日志中是否记录了敏感信息（如聊天内容、IP 地址）？产品化时应提供关闭日志或脱敏选项。

------

**输出格式**：
按上述编号逐条报告，每条包含：

1. 严重程度
2. 具体问题及触发条件
3. 可能后果
4. 修复建议（代码示例优先）
